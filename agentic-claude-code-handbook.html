    01

    Introduction to Agentic Software Engineering

    Agentic software engineering is the practice of getting reliable work out of a coding agent by shaping the system around the model, not by writing a cleverer one-off prompt each time.

    For years, AI in the editor meant autocomplete: you typed, it guessed the next line. That is a single model call with no memory and no ability to act. A coding agent is different in kind. It reads files, runs commands, inspects results, and decides what to do next — repeatedly — until a task is finished. Claude Code is exactly this: an agent that operates in your terminal or IDE with tools to read, edit, search, and execute.

    Once a model can act in a loop, the interesting engineering question stops being "what do I type?" and becomes "what does the loop see, how is it framed, what can it reach, and what is it allowed to do?" Answering those four questions well — repeatably, for a whole team — is agentic software engineering.

    The harness: everything around the model

    The model is the reasoning engine. The harness is everything the tooling team builds so the model can do useful work: dynamically assembled system prompts, tool schemas, context-gathering strategies, permission gates, and cheaper helper models for routine sub-tasks. The lesson from the GitHub Copilot guide that inspired this handbook is blunt: the enormous jump in real-world code-acceptance over the last year came far more from harness improvements than from the model alone. Better prompts, better tool design, better context strategies.

    You do not control Claude's internal harness. But every customization file you write — CLAUDE.md, a slash command, a skill, a subagent, a hook — is your layer of the harness. That is the entire game.

    The agent loop

    Strip away the mystique and the runtime is a while loop. A request comes in; the system assembles the best picture it can; the model chooses one small move; a tool runs; the result returns as fresh evidence; the loop repeats with a better picture until the task converges or is stopped.

    Fig 1.1 — The agent loop. The visible answer is the last step, not the first. Each customization primitive is a lever on one of these stages.
    ◆ The reframe that unlocks everything

    Hallucination is usually not an intelligence problem — it is a context problem. When the loop has thin evidence, the model fills gaps with plausible guesses, the same way a new hire confidently edits the wrong file if no one showed them the project first. Strengthen what the loop sees and the guesses collapse into evidence-based moves. Almost all of this handbook is about strengthening what the loop sees, frames, reaches, and is allowed to do.

    Customization is loop-shaping

    Every primitive you will meet maps to one part of the loop:

    LeverWhich part of the loop it changes
    Persistent instructionsWhat the loop sees on every turn
    Scoped / path instructionsWhat it sees when relevant files are in play
    Prompts / commands & skills & subagentsHow the task is framed
    MCP serversWhat the loop can reach
    HooksWhat the loop is allowed to do at execution time
    MemoryWhat it already knows before it starts

    Hold that table in mind. The rest of the handbook fills it in for Claude Code and shows how the pieces compose into a production framework.

    03

    Agentic Design Principles

    Before any file formats, internalize the principles. They are what let you decide which primitive to reach for and how much to put in it.

    1 · Context is the scarce resource

    The context window is finite, and every token you spend on low-signal boilerplate is a token not spent reasoning about the task. As a session grows, the harness must compact older turns into summaries, losing fidelity — which is why long sessions start to "forget" earlier decisions. Design for a lean, high-signal context, not a maximal one.

    ✓ Best practice

    Treat your CLAUDE.md like a hot cache, not a wiki. If a rule is not needed on most turns, it belongs in a scoped file, a skill, or a linked doc — not in the always-on layer.

    2 · Progressive disclosure

    The best primitives reveal detail only when needed. A skill shows the model just its name and description first; the full procedure loads only if the model judges it relevant. Mirror this everywhere: short always-on rules that point to deeper docs the agent can open on demand.

    3 · Least context, least privilege

    Give each unit of work only the context and only the tools it needs. A subagent doing read-only research should not have Write or Bash. A doc-formatting command does not need database access. Narrow scope makes behavior predictable and failures cheap.

    4 · Guidance in the middle, determinism at the edges

    Instructions influence reasoning — they are probabilistic. Hooks govern execution — they are deterministic. "Never commit secrets" in CLAUDE.md is a strong hint; a PreToolUse hook that blocks a git commit touching .env is a guarantee. Put must-nevers on the deterministic edge.

    Fig 3.1 — Choose the primitive by how strong a guarantee you need. Hints live left; enforcement lives right.

    5 · Composition over monoliths

    A real problem is rarely solved by one file type. A code-review workflow might combine an always-on style rule, a scoped test-convention file, a /review command, a security-reviewer subagent, and a hook that blocks merges on failing tests. Design the interaction, not the individual file.

    6 · Isolation makes the loop fractal

    A subagent is not a chat side-panel; it is another loop with its own context window. Delegate exploratory or parallelizable work to subagents so the main loop stays lean and never has to compact away the details that matter.

    7 · Verifiable, idempotent steps

    Prefer moves the agent can check: run the tests, re-read the file, diff the output. Prefer steps that are safe to repeat. This is what turns a hopeful generation into a self-correcting loop (Chapter 8).

    ✦ Rule of thumb

    When unsure where a piece of knowledge belongs, ask three questions in order: (a) Is it needed on almost every turn? → always-on. (b) Only for certain files or tasks? → scoped file or skill. (c) Must it be enforced, not merely encouraged? → hook or permission.

    04

    Repository Context Engineering

    This is the always-on layer — the first thing the loop sees on every turn. In Claude Code it is CLAUDE.md, and getting it right is the single highest-leverage thing you can do.

    The memory hierarchy

    Claude Code assembles its persistent context from several files, highest priority last. It also walks up the directory tree from your working directory and pulls in every CLAUDE.md it finds, and it loads a subdirectory's CLAUDE.md on demand when it starts working with files there.

    ScopeLocationUse for
    Enterprise / managedsystem managed-policy pathOrg-wide rules pushed by IT/platform teams
    Project./CLAUDE.md (repo root, checked in)Shared team conventions — the workhorse
    Directory./any/subdir/CLAUDE.mdRules that apply only inside that subtree
    User~/.claude/CLAUDE.mdYour personal defaults across all projects
    Local (legacy)./CLAUDE.local.mdDeprecated; prefer user memory or imports
    ◆ Copilot mapping

    The Copilot .github/copilot-instructions.md is always loaded, no conditions. ./CLAUDE.md is its direct equivalent. Copilot's path-specific applyTo globs are approximated in Claude Code by directory-nested CLAUDE.md files — scoped by location instead of by glob. When per-file-type rules must follow files that live anywhere, push that rule into a skill (Chapter 6) instead.

    Bootstrapping and editing

    Run /init in a fresh repo and Claude Code will scan the project and draft a starter CLAUDE.md. During any session, prefix a line with # to quickly append a memory (Claude asks which file), and run /memory to open the files in your editor.

    Imports keep it modular

    CLAUDE.md can pull in other files with @path, so a lean root file can reference deeper docs the agent reads only when relevant. Imports resolve relative or from home, and nest up to five hops.

    What belongs in the always-on layer

    Include only what is useful on most turns: the build/test/lint commands, a one-paragraph architecture map, the handful of conventions you actually enforce, hard "never do this" rules, and pointers to deeper docs. Everything else is a candidate for a scoped file, a skill, or an import.

    ✕ Anti-pattern

    The 600-line CLAUDE.md that documents every module. It burns context on every turn, buries the rules that matter, and goes stale. If you cannot skim it in thirty seconds, it is too long — split it with imports and directory-nested files.

    Prompt Engineering vs Persistent Instructions

    The same words behave completely differently depending on where you put them. The skill is knowing which of four tiers a piece of knowledge belongs to.

    TierWhereLifetimeLoads
    Ephemeral promptWhat you type in chatOne turnNow
    Reusable promptSlash command (.claude/commands/)Whenever invokedOn /name
    Conditional instructionSkill / nested CLAUDE.mdWhen relevantBy model / by location
    Persistent instructionRoot CLAUDE.mdAlwaysEvery turn
    Fig 5.1 — Decision tree: enforce → hook; always → CLAUDE.md; per-directory → nested memory; per-task → skill or command.

    Reusable prompts = slash commands

    Copilot's prompt files (*.prompt.md, invoked with /name) map exactly to Claude Code slash commands. Drop a Markdown file in .claude/commands/ (project, checked in) or ~/.claude/commands/ (personal). The filename becomes the command name. Bodies support $ARGUMENTS and positional $1, $2; can inject shell output with !\`cmd\`; and can reference files with @path.

    .claude/commands/fix-issue.md → /fix-issue 4213md
    ---
description: Reproduce, fix, and test a GitHub issue
argument-hint: <issue-number>
allowed-tools: Bash(git:*), Bash(gh:*), Read, Edit, Write
model: claude-opus-4-8
---

You are fixing issue #$1.

1. Fetch the issue:            !`gh issue view $1`
2. Current branch status:      !`git status --short`
3. Locate the relevant code, reproduce the bug with a failing test.
4. Implement the minimal fix. Re-run the test suite until green.
5. Summarize the change and the test that now covers it.

Conventions live in @CLAUDE.md — follow them exactly.
    ✦ When to make it a command vs a skill

    If you decide when to run it, make it a command (explicit, user-invoked). If you want Claude to reach for it automatically when a task matches, make it a skill (model-invoked). Same knowledge, different trigger.

    The prompt-writing discipline still applies

    Persistent files do not excuse vague prompting. Whether ephemeral or stored, good instructions are specific, name concrete constraints, give positive and negative examples, and ask for step-by-step reasoning where it helps. The difference is that a persistent instruction pays that authoring cost once and reuses it forever.

    06

    Skills and Reusable Capabilities

    A skill is a self-contained, model-invoked capability: a folder with a SKILL.md that Claude reaches for on its own when a task matches. This is the one primitive where Copilot and Claude Code use the same file format.

    The three-stage load

    Skills are the purest example of progressive disclosure. The harness discovers skill folders, but discovery is not loading:

    1. Discover — Claude Code finds skills in .claude/skills/ (project) and ~/.claude/skills/ (personal), plus any from installed plugins.
    2. Describe — the model sees only each skill's name and description. Cheap; costs almost no context.
    3. Read — only if the model judges a skill relevant does it read the full SKILL.md body (and any files it references) into context.
    ◆ The description is the trigger

    Because stage 2 is all the model sees, a vague description means the skill never loads. Name the task and the moment it applies. "Helps with documents" fails. "Use when creating, editing, or reading .docx Word files — handles headings, tables, and TOCs" fires reliably.

    Anatomy of a skill

    .claude/skills/pr-description/SKILL.mdmd
    ---
name: pr-description
description: Use when opening or updating a pull request to generate a
  structured PR description. Triggers on "write a PR description",
  "prepare this PR", or after finishing a feature branch.
---

# PR Description Generator

## Procedure
1. Run the diff against the base branch:
   `git diff --stat origin/main...HEAD`
2. Group changes by area (api / ui / infra / tests).
3. Produce the description using this exact template:

### Summary
<one sentence, imperative mood>

### Changes
- bullet per meaningful change, grouped by area

### Testing
- how it was verified; paste the passing test summary

### Risk & rollback
- blast radius and how to revert

## Notes
- Never invent test results. If tests were not run, say so.
- Keep the summary under 15 words.

    07

    Building Multi-Agent Workflows

    A subagent is not a side chat — it is another agent loop with its own context window, its own tools, and often its own model. This is how Copilot's custom agents and chat modes are recreated in Claude Code.

    Why isolation matters

    When the main loop does all the exploring — reading dozens of files, running searches, checking constraints — its context fills and starts compacting away the details that matter. Delegate that work to a subagent and the exploration happens in a separate window; only a tidy summary returns to the parent. The main loop stays lean and sharp. Isolation is the whole point.

    Fig 7.1 — Orchestrator-workers. Each subagent runs an isolated loop with least-privilege tools and hands back a compact result.

    Defining a subagent

    Subagents live in .claude/agents/*.md (project) or ~/.claude/agents/*.md (personal). Frontmatter sets identity and — crucially — its tools and model. The body is that agent's system prompt. Claude delegates automatically when the description matches, or you can name it explicitly ("use the security-reviewer subagent").

    .claude/agents/security-reviewer.mdmd
    ---
name: security-reviewer
description: Use PROACTIVELY after code changes to review for security
  issues. Focuses on the OWASP Top 10. Read-only; never edits code.
tools: Read, Grep, Glob
model: claude-opus-4-8
---

You are a security reviewer. You do not fix code — you assess it.

1. Begin every review with a one-line risk assessment (low/med/high).
2. Walk the diff for: injection, authz gaps, secret handling,
   unsafe deserialization, SSRF, and dependency risk.
3. For each finding: file:line, severity, why it matters, and a
   suggested direction (not a full patch).
4. End with a go / no-go recommendation.

Do not propose fixes until the full review is complete.

    Recreating Copilot "chat modes"

    A Copilot chat mode bundles a persona + a tool set + behavior. Claude Code splits that across two primitives, and you pick by scope:

    You want to change…Use
    Behavior for a delegated task (own context/tools/model)A subagent
    Claude's overall persona/output for the whole session (e.g. "explain like a teacher")An output style (/output-style, files in .claude/output-styles/)
    A repeatable task entry pointA slash command that can hand off to a subagent
    ✓ Best practice

    Give research/review subagents read-only tools. It makes their behavior predictable, prevents accidental edits during exploration, and lets you run several in parallel without them stepping on each other.

    08

    Planning, Execution & Validation Loops

    A good agentic workflow is a closed loop: plan → execute → verify → correct. The magic is not generation; it is the verify step feeding evidence back in.

    Fig 8.1 — The self-correcting loop. A failed verification is not a dead end; it is the best possible next-turn context.

    Plan before code

    Claude Code has a built-in plan mode — it investigates and proposes a plan without touching files until you approve. Toggle it with Shift+Tab, or start a session in it. For anything non-trivial, planning first prevents the agent from charging into the wrong design.

    terminal — start in plan modebash
    # Investigate & propose a plan, make no edits until approved
claude --permission-mode plan

# The canonical workflow, spoken to the agent:
> Explore the auth module and the failing test. Do NOT write code yet.
> Propose a plan. I will approve before you implement.

    Persist the plan as an artifact

    For multi-session work, have the agent write the plan to a durable file — a PLAN.md, a task list, or a spreadsheet task board. This survives context compaction and lets you (or a fresh session) resume exactly where you left off. A checked-in plan doc is also the natural place for a subagent to report progress.

    Make verification deterministic

    Two layers make the loop trustworthy. First, explicit verify steps in your prompts, commands, and skills ("re-run the suite until green; paste the summary"). Second, hooks that run checks automatically after edits, so the agent cannot skip them.

    .claude/settings.json — auto-format & lint after every editjson
    {
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "make fmt && make lint" }
        ]
      }
    ]
  }
}
    ✦ Test-driven agentic loop

    Ask for the failing test first, confirm it fails, then implement until it passes. A concrete red/green signal is far stronger context than any amount of prose about "correct behavior," and it turns the agent's own tool output into its guide.

    ✕ Anti-pattern

    "Generate the whole feature, then review at the end." Long unbroken generations accumulate errors with no feedback, fill context, and force compaction. Decompose into verify-able steps and let each result inform the next move.

    09

    Translating GitHub Copilot Concepts to Claude Code

    This is the core adaptation. For each primitive: the problem it solves, whether Claude Code has an equivalent, how to recreate it if the shape differs, and the practical workflow. Everything here runs inside your Claude Code subscription — no Claude API needed.

    ▲ On the "no API" constraint

    You never call the API directly. Claude Code is the harness and the runtime. Slash commands, skills, subagents, hooks, and MCP servers are all read from your repo and home directory by the Claude Code process. MCP servers you add run as local (or remote) tools the agent calls — again, no API key of your own required for any of the customization below.

    Fig 9.1 — Six of eight primitives have a direct or identical counterpart. Only two — path-glob instructions and cloud-learned memory — are approximated by composing Claude Code primitives.
    P1Always-on instructions → CLAUDE.md

    Problem: baseline rules the agent should honor on every turn without being reminded.

    Equivalent? Yes, direct. ./CLAUDE.md is loaded unconditionally, exactly like copilot-instructions.md.

    Workflow: /init to draft it, then curate to the essentials (commands, architecture, conventions, hard nevers). Keep it skimmable; push depth into @imports.

    P2File-based instructions → nested memory + skills

    Problem: rules that should apply only to certain files (e.g. "every route validates with Zod").

    Equivalent? Partial. Claude Code has no applyTo glob on instruction files. It scopes by directory: a CLAUDE.md inside src/routes/ loads when the agent works there.

    Recreate: for location-based rules, use nested CLAUDE.md. For rules that must follow a file type wherever it lives, encode the rule as a skill whose description names that file type — the model loads it when the task matches.

    P3Prompts → slash commands

    Problem: a reusable, parameterized prompt you invoke on demand.

    Equivalent? Yes, direct. .claude/commands/name.md → /name, with $ARGUMENTS/$1, !\`cmd\` shell injection, @file refs, and frontmatter for allowed-tools/model.

    Workflow: keep team commands in the repo; personal ones in ~/.claude/commands/. Organize with subfolders.

    P4Skills → Agent Skills

    Problem: a capability the agent should self-select when a task matches.

    Equivalent? Yes — identical file format. .claude/skills/<name>/SKILL.md with a name+description frontmatter and progressive three-stage loading.

    Workflow: invest in the description; bundle deterministic scripts for exact steps; link reference docs the body reads on demand.

    P5Custom agents / chat modes → subagents + output styles

    Problem: a distinct persona with its own tools/behavior, or a whole-session mode.

    Equivalent? Yes. Delegated personas → subagents (.claude/agents/*.md, own context/tools/model). Whole-session persona → output styles.

    Workflow: read-only tools for reviewers/researchers; let the main loop orchestrate and delegate.

    P6MCP → MCP

    Problem: reach beyond the local workspace — databases, issue trackers, internal APIs.

    Equivalent? Yes — same open protocol. Configure servers in .mcp.json (project, checked in) or via claude mcp add. Tools appear to the agent and execute when it calls them.

    Workflow: scope servers per project; prefer read-only credentials; see Chapter 16.

    P7Hooks → hooks

    Problem: deterministic control at execution boundaries (format on save, block dangerous commands).

    Equivalent? Yes. Configure in .claude/settings.json under lifecycle events: PreToolUse, PostToolUse, UserPromptSubmit, Stop, and more. A non-zero PreToolUse exit can block a tool call.

    Workflow: put must-nevers here, not in prose. Keep hook commands fast.

    P8Memory → CLAUDE.md + quick-add + /memory

    Problem: knowledge that accrues as the team works in the repo.

    Equivalent? Approximated. Claude Code has no automatic cloud-learned memory; instead memory is explicit and version-controlled — which is arguably better for auditability.

    Recreate: when a useful fact emerges, capture it with # (quick-add) or curate via /memory. Because it lives in CLAUDE.md, it is reviewable in PRs and shared with the whole team.

    Goal: stand up your own MCP server exposing two safe, domain-relevant tools — list listening sockets and probe an HTTP endpoint — then wire it into the repo so the agent can call real system observability without shelling out. This is P6 (MCP) end-to-end.

    #!/usr/bin/env python3
"""sysobs-mcp - a minimal, READ-ONLY system-observability MCP server.

Exposes two tools over stdio:
  open_ports     : list listening TCP sockets (`ss -ltn`)
  probe_endpoint : HTTP GET a URL, return status + a few safe headers

No shell is used (subprocess with an argument list), and no tool mutates
state. Requires Python 3.10+ and the `mcp` package (`pip install mcp`).
"""
from __future__ import annotations

import json
import subprocess
import urllib.request
import urllib.error

from mcp.server.fastmcp import FastMCP

# FastMCP builds each tool's schema from the type hints + docstring below.
mcp = FastMCP("sysobs")

_KEEP_HEADERS = ("content-type", "server", "location", "content-length")


@mcp.tool()
def open_ports() -> str:
    """List listening TCP sockets on this host (read-only; runs `ss -ltn`)."""
    try:
        proc = subprocess.run(
            ["ss", "-ltn"],           # argument list, never a shell string
            capture_output=True,
            text=True,
            timeout=5,
            check=False,
        )
        return proc.stdout or "(none)"
    except FileNotFoundError:
        return "error: `ss` not found (install iproute2)"
    except subprocess.TimeoutExpired:
        return "error: `ss` timed out"


@mcp.tool()
def probe_endpoint(url: str) -> str:
    """HTTP GET a URL and return status code + selected headers as JSON.

    Read-only: no request body is sent. Use to smoke-test a REST endpoint.
    """
    if not url.startswith(("http://", "https://")):
        return "error: url must start with http:// or https://"

    req = urllib.request.Request(url, method="GET")
    try:
        with urllib.request.urlopen(req, timeout=5) as resp:
            headers = {h: resp.headers.get(h) for h in _KEEP_HEADERS if resp.headers.get(h)}
            out = {"status": resp.status, "ok": 200 <= resp.status < 300, "headers": headers}
            return json.dumps(out, indent=2)
    except urllib.error.HTTPError as e:
        headers = {h: e.headers.get(h) for h in _KEEP_HEADERS if e.headers.get(h)}
        return json.dumps({"status": e.code, "ok": False, "headers": headers}, indent=2)
    except Exception as e:  # report any transport failure as tool text
        return f"probe failed: {e}"


if __name__ == "__main__":
    mcp.run()  # stdio transport by default

    Best Practices

    The distilled operating rules. Most trace back to one idea: put each piece of knowledge on the stage of the loop where it does the most good for the least context.

    ✓ Keep the always-on layer lean

    A skimmable CLAUDE.md beats an exhaustive one. Every token there is paid on every turn. Push depth into @imports, skills, and directory-nested files.

    ✓ Write skill descriptions as triggers

    Name the task and the moment. The description is all the model sees before deciding to load — treat it like a function signature, not a tagline.

    ✓ Least privilege everywhere

    Read-only tools for researchers and reviewers. Tight allow/deny permission lists. Read-only DB credentials for MCP. Narrow scope = predictable behavior.

    ✓ Enforce with hooks, not prose

    If it must never happen, a PreToolUse hook or a deny rule guarantees it. Instructions only nudge.

    ✓ Plan first, verify always

    Use plan mode for non-trivial work. Close every loop with a concrete check — a passing test, a re-read, a diff. Prefer TDD's red/green signal over prose about correctness.

    ✓ Decompose and delegate

    Offload exploration to subagents so the main loop stays lean and never has to compact away what matters.

    ✓ Prefer scripts for deterministic steps

    Bundle a script in a skill instead of describing an algorithm. The agent runs exact code rather than re-deriving it each time.

    ✓ Commit and iterate on the harness

    Version your customization files, review them in PRs, and improve them when the agent stumbles. Your harness should get better faster than the model does.

    Weekly harness hygiene

    • Trim anything stale or unused from CLAUDE.md
    • Note any recurring correction and encode it as a rule, skill, or hook
    • Check that skills you expected to fire actually did (tighten descriptions)
    • Confirm hooks still match the tools they intend to guard

    Common Mistakes

    Nearly every one is a loading mistake — a primitive placed where the loop can't use it as intended.

    ✕ The kitchen-sink CLAUDE.md

    Hundreds of lines documenting every module. It buries the rules that matter and taxes every turn. Fix: keep essentials; move the rest to imports and nested files.

    ✕ Vague skill descriptions

    "Helps with code." The model never matches it, so it never loads. Fix: name the task and the trigger words.

    ✕ Must-nevers only in prose

    "Never touch production" in CLAUDE.md is a hint the agent can rationalize past. Fix: a deny rule + PreToolUse hook.

    ✕ Hook watching the wrong tool

    A hook on Write won't fire when the model runs a shell command. Fix: match the actual tool (Bash, Edit, etc.) and test it.

    ✕ Subagents with write access "to be safe"

    A research agent that can edit will eventually edit. Fix: read-only tools for anything exploratory.

    ✕ Doing everything in the main loop

    Reading 40 files inline fills context and forces lossy compaction, so the agent "forgets" earlier decisions. Fix: delegate exploration to subagents.

    ✕ Confusing commands and skills

    Making a /command for something you want auto-selected, or a skill for something you always dispatch by hand. Fix: you-trigger → command; model-triggers → skill.

    ✕ Expecting invisible cloud memory

    Claude Code won't silently "learn" your repo across sessions. Fix: capture facts explicitly with # / /memory — and enjoy that they're now reviewable.

    ✕ Over-customizing before there's a problem

    Building ten agents and twenty commands on day one. Fix: start with a good CLAUDE.md; add a primitive only when a real, repeated pain justifies it.

    ✕ Not versioning the harness

    Config living only on one laptop. Fix: commit it; a bad instruction is a reviewable, revertable change like any other.

    #!/usr/bin/env python3
"""cppkit-mcp - build & diagnostics tools for a CMake C/C++ project.

Exposes typed tools over stdio so the agent reasons over STRUCTURED
diagnostics instead of raw compiler stderr:
  configure  : run CMake configure (also exports compile_commands.json)
  build      : compile and return parsed {file,line,col,severity,message}
  run_tests  : run ctest and return pass/fail with the failing output

All tools run in the current working directory using subprocess argument
lists (no shell). Requires Python 3.10+, the `mcp` package, and a CMake +
compiler toolchain on PATH.
"""
from __future__ import annotations

import json
import re
import subprocess

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("cppkit")

# A gcc/clang diagnostic line:  path:line:col: severity: message
_DIAG = re.compile(
    r"^(?P<file>[^:\n]+):(?P<line>\d+):(?P<col>\d+):\s+"
    r"(?P<severity>error|warning|note):\s+(?P<message>.*)$"
)


def _run(args: list[str], timeout: int = 180) -> subprocess.CompletedProcess[str]:
    return subprocess.run(
        args, capture_output=True, text=True, timeout=timeout, check=False
    )


@mcp.tool()
def configure(build_dir: str = "build") -> str:
    """Run CMake configure with compile_commands.json export. Returns JSON."""
    try:
        cp = _run(["cmake", "-S", ".", "-B", build_dir,
                   "-DCMAKE_EXPORT_COMPILE_COMMANDS=ON"])
    except FileNotFoundError:
        return json.dumps({"ok": False, "error": "cmake not found on PATH"})
    except subprocess.TimeoutExpired:
        return json.dumps({"ok": False, "error": "cmake configure timed out"})
    tail = (cp.stdout + cp.stderr).strip().splitlines()[-12:]
    return json.dumps(
        {"ok": cp.returncode == 0, "returncode": cp.returncode, "tail": tail},
        indent=2,
    )


@mcp.tool()
def build(build_dir: str = "build", target: str | None = None) -> str:
    """Compile the project and return parsed diagnostics as structured JSON."""
    args = ["cmake", "--build", build_dir]
    if target:
        args += ["--target", target]
    try:
        cp = _run(args)
    except FileNotFoundError:
        return json.dumps({"ok": False, "error": "cmake not found on PATH"})
    except subprocess.TimeoutExpired:
        return json.dumps({"ok": False, "error": "build timed out"})

    diags = []
    for line in (cp.stdout + "\n" + cp.stderr).splitlines():
        m = _DIAG.match(line.strip())
        if m:
            d = m.groupdict()
            d["line"] = int(d["line"])
            d["col"] = int(d["col"])
            diags.append(d)

    return json.dumps({
        "ok": cp.returncode == 0,
        "returncode": cp.returncode,
        "error_count": sum(1 for d in diags if d["severity"] == "error"),
        "warning_count": sum(1 for d in diags if d["severity"] == "warning"),
        "diagnostics": diags,
    }, indent=2)


@mcp.tool()
def run_tests(build_dir: str = "build") -> str:
    """Run ctest and return pass/fail plus the failing output tail."""
    try:
        cp = _run(["ctest", "--test-dir", build_dir, "--output-on-failure"])
    except FileNotFoundError:
        return json.dumps({"ok": False, "error": "ctest not found on PATH"})
    except subprocess.TimeoutExpired:
        return json.dumps({"ok": False, "error": "tests timed out"})
    tail = (cp.stdout + cp.stderr).strip().splitlines()[-20:]
    return json.dumps(
        {"ok": cp.returncode == 0, "returncode": cp.returncode, "output_tail": tail},
        indent=2,
    )


if __name__ == "__main__":
    mcp.run()  # stdio transport by default

    The dockyard MCP server (Python)

    mcp/dockyard_server.py (complete)python
    #!/usr/bin/env python3
"""dockyard-mcp - drive a 3-container C/C++ pipeline through Docker.

Wraps `docker exec` against three FIXED containers so agents can build,
unit-test/debug, and deploy without arbitrary container or shell access:
  configure -> BuildDockerContainer    (cmake configure)
  build     -> BuildDockerContainer    (cross-compile; structured diagnostics)
  unit_test -> TargetTestRunContainer  (ctest; pass/fail)
  debug     -> TargetTestRunContainer  (run a built binary under an allowed tool)
  deploy    -> deploymentContainer     (staging by default; prod is gated)

Only the three named containers are reachable; commands run as argument lists
(no shell). Requires Python 3.10+, `mcp`, and docker on PATH.
"""
from __future__ import annotations

import json
import re
import subprocess

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("dockyard")

BUILD_C = "BuildDockerContainer"
TARGET_C = "TargetTestRunContainer"
DEPLOY_C = "deploymentContainer"
WORKDIR = "/workspace"          # shared bind mount in every container

_DIAG = re.compile(
    r"^(?P<file>[^:\n]+):(?P<line>\d+):(?P<col>\d+):\s+"
    r"(?P<severity>error|warning|note):\s+(?P<message>.*)$"
)

# allow-lists keep the surface small
_DEBUG_TOOLS = {
    "run": [],
    "valgrind": ["--error-exitcode=1", "--leak-check=full"],
    "gdb": ["-batch", "-ex", "run", "-ex", "bt"],
}
_DEPLOY_ENVS = {"staging"}      # prod requires confirm=True (see deploy)


def _dexec(container: str, argv: list[str], timeout: int = 300):
    return subprocess.run(
        ["docker", "exec", "-w", WORKDIR, container, *argv],
        capture_output=True, text=True, timeout=timeout, check=False,
    )


def _diagnostics(text: str) -> list[dict]:
    out = []
    for line in text.splitlines():
        m = _DIAG.match(line.strip())
        if m:
            d = m.groupdict()
            d["line"] = int(d["line"])
            d["col"] = int(d["col"])
            out.append(d)
    return out


@mcp.tool()
def configure() -> str:
    """CMake configure inside BuildDockerContainer (+ compile_commands.json)."""
    try:
        cp = _dexec(BUILD_C, ["cmake", "-S", ".", "-B", "build",
                              "-DCMAKE_EXPORT_COMPILE_COMMANDS=ON"])
    except FileNotFoundError:
        return json.dumps({"ok": False, "error": "docker not found on PATH"})
    except subprocess.TimeoutExpired:
        return json.dumps({"ok": False, "error": "configure timed out"})
    tail = (cp.stdout + cp.stderr).strip().splitlines()[-12:]
    return json.dumps({"container": BUILD_C, "ok": cp.returncode == 0,
                       "returncode": cp.returncode, "tail": tail}, indent=2)


@mcp.tool()
def build(target: str | None = None) -> str:
    """Cross-compile inside BuildDockerContainer; return structured diagnostics."""
    argv = ["cmake", "--build", "build"]
    if target:
        argv += ["--target", target]
    try:
        cp = _dexec(BUILD_C, argv)
    except FileNotFoundError:
        return json.dumps({"ok": False, "error": "docker not found on PATH"})
    except subprocess.TimeoutExpired:
        return json.dumps({"ok": False, "error": "build timed out"})
    diags = _diagnostics(cp.stdout + "\n" + cp.stderr)
    return json.dumps({
        "container": BUILD_C,
        "ok": cp.returncode == 0,
        "returncode": cp.returncode,
        "error_count": sum(1 for d in diags if d["severity"] == "error"),
        "warning_count": sum(1 for d in diags if d["severity"] == "warning"),
        "diagnostics": diags,
    }, indent=2)


@mcp.tool()
def unit_test() -> str:
    """Run ctest inside TargetTestRunContainer; return pass/fail + output tail."""
    try:
        cp = _dexec(TARGET_C, ["ctest", "--test-dir", "build",
                               "--output-on-failure"])
    except FileNotFoundError:
        return json.dumps({"ok": False, "error": "docker not found on PATH"})
    except subprocess.TimeoutExpired:
        return json.dumps({"ok": False, "error": "tests timed out"})
    tail = (cp.stdout + cp.stderr).strip().splitlines()[-20:]
    return json.dumps({"container": TARGET_C, "ok": cp.returncode == 0,
                       "returncode": cp.returncode, "output_tail": tail}, indent=2)


@mcp.tool()
def debug(program: str, tool: str = "run") -> str:
    """Run a built program in TargetTestRunContainer under an allowed tool.

    tool is one of {run, valgrind, gdb}. `program` is a path under build/.
    """
    if tool not in _DEBUG_TOOLS:
        return json.dumps({"ok": False,
                           "error": "tool must be one of run, valgrind, gdb"})
    if "/" not in program:
        program = "build/" + program
    if ".." in program:
        return json.dumps({"ok": False, "error": "invalid program path"})
    argv = [program] if tool == "run" else [tool, *_DEBUG_TOOLS[tool], program]
    try:
        cp = _dexec(TARGET_C, argv, timeout=180)
    except subprocess.TimeoutExpired:
        return json.dumps({"ok": False, "error": "debug run timed out"})
    tail = (cp.stdout + cp.stderr).strip().splitlines()[-40:]
    return json.dumps({"container": TARGET_C, "tool": tool,
                       "ok": cp.returncode == 0, "returncode": cp.returncode,
                       "output_tail": tail}, indent=2)


@mcp.tool()
def deploy(artifact: str, env: str = "staging", confirm: bool = False) -> str:
    """Deploy an artifact via deploymentContainer. Prod requires confirm=True."""
    if ".." in artifact or artifact.startswith("/"):
        return json.dumps({"ok": False,
                           "error": "artifact must be a workspace-relative path"})
    if env not in _DEPLOY_ENVS and not (env == "prod" and confirm):
        return json.dumps({"ok": False,
                           "error": "env not allowed (prod needs confirm=true)"})
    try:
        cp = _dexec(DEPLOY_C, ["/opt/deploy.sh", env, artifact])
    except subprocess.TimeoutExpired:
        return json.dumps({"ok": False, "error": "deploy timed out"})
    tail = (cp.stdout + cp.stderr).strip().splitlines()[-20:]
    return json.dumps({"container": DEPLOY_C, "env": env,
                       "ok": cp.returncode == 0, "returncode": cp.returncode,
                       "output_tail": tail}, indent=2)

#include <string>
#include <utility>
#include <variant>

namespace rb {

struct Error {
    std::string message;
};

template <typename T>
class Result {
public:
    Result(T value) : data_(std::move(value)) {}
    Result(Error error) : data_(std::move(error)) {}

    [[nodiscard]] bool ok() const noexcept {
        return std::holds_alternative<T>(data_);
    }

    mcp/dockyard_server.py (complete)python
    #!/usr/bin/env python3
"""dockyard-mcp - drive the 3-container pipeline through Docker.

Tools (stdio), each routed to a fixed container via `docker exec`:
  configure -> BuildDockerContainer     (cmake configure)
  build     -> BuildDockerContainer     (compile; structured diagnostics)
  unit_test -> TargetTestRunContainer   (ctest; pass/fail)
  debug     -> TargetTestRunContainer   (run a built binary under an allowed tool)
  deploy    -> deploymentContainer      (/opt/deploy.sh; staging default, prod gated)

Argument lists only (no shell). Requires Python 3.10+, `mcp`, docker on PATH.
"""
from __future__ import annotations

import json
import re
import subprocess

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("dockyard")

BUILD_C = "BuildDockerContainer"
TARGET_C = "TargetTestRunContainer"
DEPLOY_C = "deploymentContainer"
WORKDIR = "/workspace"

_DIAG = re.compile(
    r"^(?P<file>[^:\n]+):(?P<line>\d+):(?P<col>\d+):\s+"
    r"(?P<severity>error|warning|note):\s+(?P<message>.*)$"
)

_DEBUG_TOOLS = {
    "run": [],
    "valgrind": ["--error-exitcode=1", "--leak-check=full"],
    "gdb": ["-batch", "-ex", "run", "-ex", "bt"],
}
_DEPLOY_ENVS = {"staging"}


def _dexec(container: str, argv: list[str], timeout: int = 300):
    return subprocess.run(
        ["docker", "exec", "-w", WORKDIR, container, *argv],
        capture_output=True, text=True, timeout=timeout, check=False,
    )


def _diagnostics(text: str) -> list[dict]:
    out = []
    for line in text.splitlines():
        m = _DIAG.match(line.strip())
        if m:
            d = m.groupdict()
            d["line"] = int(d["line"])
            d["col"] = int(d["col"])
            out.append(d)
    return out


@mcp.tool()
def configure() -> str:
    """CMake configure inside BuildDockerContainer (+ compile_commands.json)."""
    try:
        cp = _dexec(BUILD_C, ["cmake", "-S", ".", "-B", "build",
                              "-DCMAKE_EXPORT_COMPILE_COMMANDS=ON"])
    except FileNotFoundError:
        return json.dumps({"ok": False, "error": "docker not found on PATH"})
    except subprocess.TimeoutExpired:
        return json.dumps({"ok": False, "error": "configure timed out"})
    tail = (cp.stdout + cp.stderr).strip().splitlines()[-12:]
    return json.dumps({"container": BUILD_C, "ok": cp.returncode == 0,
                       "returncode": cp.returncode, "tail": tail}, indent=2)


@mcp.tool()
def build(target: str | None = None) -> str:
    """Compile inside BuildDockerContainer; return structured diagnostics."""
    argv = ["cmake", "--build", "build"]
    if target:
        argv += ["--target", target]
    try:
        cp = _dexec(BUILD_C, argv)
    except FileNotFoundError:
        return json.dumps({"ok": False, "error": "docker not found on PATH"})
    except subprocess.TimeoutExpired:
        return json.dumps({"ok": False, "error": "build timed out"})
    diags = _diagnostics(cp.stdout + "\n" + cp.stderr)
    return json.dumps({
        "container": BUILD_C,
        "ok": cp.returncode == 0,
        "returncode": cp.returncode,
        "error_count": sum(1 for d in diags if d["severity"] == "error"),
        "warning_count": sum(1 for d in diags if d["severity"] == "warning"),
        "diagnostics": diags,
    }, indent=2)


@mcp.tool()
def unit_test() -> str:
    """Run ctest inside TargetTestRunContainer; return pass/fail + output tail."""
    try:
        cp = _dexec(TARGET_C, ["ctest", "--test-dir", "build",
                               "--output-on-failure"])
    except FileNotFoundError:
        return json.dumps({"ok": False, "error": "docker not found on PATH"})
    except subprocess.TimeoutExpired:
        return json.dumps({"ok": False, "error": "tests timed out"})
    tail = (cp.stdout + cp.stderr).strip().splitlines()[-20:]
    return json.dumps({"container": TARGET_C, "ok": cp.returncode == 0,
                       "returncode": cp.returncode, "output_tail": tail}, indent=2)


@mcp.tool()
def debug(program: str, tool: str = "run") -> str:
    """Run a built program in TargetTestRunContainer under an allowed tool.

    tool is one of {run, valgrind, gdb}. `program` is a path under build/.
    """
    if tool not in _DEBUG_TOOLS:
        return json.dumps({"ok": False,
                           "error": "tool must be one of run, valgrind, gdb"})
    if "/" not in program:
        program = "build/" + program
    if ".." in program:
        return json.dumps({"ok": False, "error": "invalid program path"})
    argv = [program] if tool == "run" else [tool, *_DEBUG_TOOLS[tool], program]
    try:
        cp = _dexec(TARGET_C, argv, timeout=180)
    except subprocess.TimeoutExpired:
        return json.dumps({"ok": False, "error": "debug run timed out"})
    tail = (cp.stdout + cp.stderr).strip().splitlines()[-40:]
    return json.dumps({"container": TARGET_C, "tool": tool,
                       "ok": cp.returncode == 0, "returncode": cp.returncode,
                       "output_tail": tail}, indent=2)


@mcp.tool()
def deploy(artifact: str, env: str = "staging", confirm: bool = False) -> str:
    """Deploy an artifact via deploymentContainer. Prod requires confirm=True."""
    if ".." in artifact or artifact.startswith("/"):
        return json.dumps({"ok": False,
                           "error": "artifact must be a workspace-relative path"})
    if env not in _DEPLOY_ENVS and not (env == "prod" and confirm):
        return json.dumps({"ok": False,
                           "error": "env not allowed (prod needs confirm=true)"})
    try:
        cp = _dexec(DEPLOY_C, ["/opt/deploy.sh", env, artifact])
    except subprocess.TimeoutExpired:
        return json.dumps({"ok": False, "error": "deploy timed out"})
    tail = (cp.stdout + cp.stderr).strip().splitlines()[-20:]
    return json.dumps({"container": DEPLOY_C, "env": env,
                       "ok": cp.returncode == 0, "returncode": cp.returncode,
                       "output_tail": tail}, indent=2)

    Project servers from .mcp.json are disabled until approved — by design, so a cloned repo can't auto-run code. While pending, mcp__dockyard__* doesn't exist and Claude may fall back to raw docker exec (which the deny rule above now blocks). Walk these in order:

    1 · Approve it. Run /mcp in-session; dockyard shows its status and an approve action. claude mcp list shows a pending project server as ⏸ Pending approval.

    2 · Trust the workspace. Recent Claude Code ignores checked-in approvals (and enableAllProjectMcpServers) until you run claude in the folder and accept the workspace-trust dialog. A freshly cloned repo can't approve its own servers.

    3 · Reset if you dismissed the prompt. If you accidentally denied it, claude mcp reset-project-choices re-triggers approval on next launch.

    4 · Launch from the repo root. The command path ./mcp/.venv/bin/python is relative; starting claude elsewhere breaks it. Keep .mcp.json at the project root (not under .claude/).

    5 · Test the server standalone. ./mcp/.venv/bin/python ./mcp/dockyard_server.py should start and wait on stdio without crashing; if it exits, fix that (missing mcp dep, wrong interpreter) — a server that crashes on start shows as disconnected.

    Seeing -32000 / "failed to reconnect"? Different failure: the server process died on launch or during the handshake — not slowness. Take Claude out of the loop and run the exact command by hand: ./mcp/.venv/bin/python ./mcp/dockyard_server.py. A Python traceback names the cause (missing mcp dep, Python < 3.10, wrong interpreter); if it sits silently waiting on stdin, the server is fine and the problem is the launch path. The most common stdio cause is the relative command path — prefer absolute: claude mcp add dockyard --scope project -- "$(realpath mcp/.venv/bin/python)" "$(realpath mcp/dockyard_server.py)". Never print() to stdout in the server — stdout carries the JSON-RPC stream; log to stderr. For the exact stderr, run claude --debug (or claude doctor).

    pip says "No matching distribution … mcp (from versions: none)"? The mcp SDK needs Python 3.10+. On Ubuntu 20.04 (which many build VMs run) python3 is 3.8, so pip filters out every wheel and reports none. Check mcp/.venv/bin/python --version; if it's < 3.10, install python3.11 (deadsnakes PPA) or use uv run --python 3.11 --with mcp …, rebuild the venv, and point .mcp.json at it.     Future Extensions

    Where to grow the framework next: reaching real systems (MCP), richer memory, and treating your harness as software you evaluate.

    MCP beyond the basics

    MCP is the loop's window to the outside world. Beyond local npx servers, you can attach remote/HTTP servers, browse the growing ecosystem of prebuilt connectors, and gate them with auth. The security discipline is the same as everywhere else in this handbook: least privilege.

    Memory systems

    Because Claude Code memory is just files, you can build richer schemes on top: a curated docs/decisions/ log the agent greps, a structured task board it reads and updates, or a lightweight index of internal knowledge exposed through a filesystem MCP server. The pattern is always the same — make durable knowledge reachable and let the loop pull it on demand.

    Treat the harness as software

    The most durable extension is a mindset: your commands, skills, agents, and instructions are a codebase. Keep a small set of "golden" tasks and periodically check the agent still handles them well after harness changes. This is the lightweight version of the evaluation discipline that made the underlying tooling improve so fast in the first place.

    The ecosystem is converging

    Skills (SKILL.md) and MCP are already shared conventions across tools; instruction files (AGENTS.md, CLAUDE.md) are increasingly cross-read. Investing in these primitives is a bet that pays off no matter which agent you run tomorrow.
